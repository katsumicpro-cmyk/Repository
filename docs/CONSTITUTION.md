# Innovation OS — Constitution

> この文書はInnovation OSが何であり、何でないかを定義する。
> コードより先に読む。設計判断より先に参照する。

---

## Purpose

知識を保存するシステムを作らない。
知識が育つシステムを作る。

Innovation OS の目的は、
人間とAIが共同で「理解を深める」プロセスを
インフラストラクチャとして支えることである。

---

## Vision

発見が知識になり、
知識がパターンを見つけ、
パターンが原則を導き、
原則が未来を描き、
未来がConceptになる。

このサイクルが止まらずに回り続けるシステム。

---

## North Star

**「システムが人間より先に問いを立てる」**

これが達成されたとき、Innovation OS は
情報管理ツールではなく、
知的パートナーになる。

---

## Design Principles

### 1. 能力として定義する

Sprint の完了条件は「コードが完成した」ではない。
**「新しい能力が獲得された」** である。

能力は:
- 動詞で表現できる（記憶できる / 想起できる / 問いを立てる）
- コードを読まずに示せる
- そのSprintの前には不可能で、後には可能になっている
- 他の能力の土台になりうる

コードは能力を実現する手段であり、目的ではない。

### 2. 知識は動く

知識を保存することは目的ではない。
知識が矛盾を発見し、パターンを抽出し、仮説を生み、問いを立てる。
**知識は動詞でなければならない。**

静的なデータベースを作るなら、他に良いツールがある。

### 3. 構造が先、AIが後

AIを呼ぶ前に、ドメインモデルを正しく設計する。
AIがなくても動くシステムが、AIを加えてより賢くなる。
AIに依存するシステムは、AIが変わるたびに壊れる。

### 4. 境界は知識の文法である

パッケージの境界はコードの整理ではない。
**知識の文法**である。

`discovery` は `knowledge` を知らない。
`knowledge` は `application` を知らない。
これは技術的制約ではなく、**概念の分離**である。

### 5. 問いは成果物である

`EvidenceRequest` はシステムの出力の一つである。
コードが増えることよりも、
システムが立てる問いが鋭くなることの方が、進歩の証拠だ。

---

## AI Principles

### AIはドメインを知らない

`packages/domain` に AI SDK を import しない。
AIはPortを通じてのみ呼ばれる。
ドメインモデルはAIが変わっても変わらない。

### AIは検証ではなく、探索に使う

AIが「正しい答え」を返すことを前提にしない。
AIが返したものは `Fact` として記録され、
`ConfidenceScore` が付与され、
矛盾検出の対象になる。
AIも知識の一つの情報源に過ぎない。

### AIより先にシステムが問いを立てる

AIにクエリを投げる前に、
システム自身が何を問うべきかを知っている状態を目指す。

---

## Knowledge Principles

### Factは事実の記録である

`KnowledgeFact` は主張ではなく、観察の記録である。
確信度はその観察への信頼度であり、真偽の宣言ではない。

### 矛盾は削除しない

矛盾する事実が見つかったとき、どちらかを消さない。
矛盾は `Contradiction` として記録され、
解消のための `EvidenceRequest` が生成される。
矛盾は知識の欠落ではなく、**未解決の問い**である。

### 知識は時間とともに変化する

昨日の高確信度事実は、今日の低確信度事実になりうる。
システムはこの変化を追跡し、
「以前は確かだと思っていたことが揺らいでいる」を
検出できなければならない。

---

## Architecture Principles

```
Presentation → Application → Domain → Shared
                    ↓
              Infrastructure
```

依存の方向は内側へのみ。
外側が内側を知ってよい。内側が外側を知ってはならない。

Build Order: `shared → foundation → domain → knowledge → application → infrastructure → presentation`

UIから実装を始めない。能力から設計し、Domainから実装する。

---

## Decision Principles

### 決定は記録する

`docs/DECISIONS.md` に残す。
「なぜそう決めたか」が消えると、
将来の設計者は同じ間違いを繰り返す。

### 制約は理由とともに存在する

「AIを呼ばない」には理由がある。
「Supabaseを使わない」には段階がある。
理由のない制約は守られない。

### 5年後の設計者に説明できるか

実装の前に問う: 5年後、このコードを初めて読む人が
なぜこう書かれているかを理解できるか。
理解できない設計は、今の自分にも理解できていない。
