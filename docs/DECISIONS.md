# Architecture Decisions

軽量な設計判断ログ。  
大規模な決定は `docs/adr/` に ADR として昇格させる。

## D001 — Knowledge First

**日付**: 2026-06-26  
**状態**: 採用

Knowledge Engine を Pattern Engine より先に実装する。

**理由**  
Knowledgeが存在しなければ Pattern品質は向上しない。  
AIがパターンを抽出する前に、  
Factの構造化・関係付け・クラスタリングの基盤が必要である。

**影響**  
Sprint 3 は `packages/knowledge` の実装から開始する。

## D002 — Discovery は検索ではない

**日付**: 2026-06-26  
**状態**: 採用

Discovery は Knowledge生成の入口である。

**理由**  
「情報を見つける」行為と「知識を生み出す」行為は異なる。  
Discovery ドメインの責務は、生の情報 (Fact) を  
構造化された知識の種へ変換することである。  
検索エンジンの代替ではない。

## D003 — Project Memory を Repository の正式成果物とする

**日付**: 2026-06-26  
**状態**: 採用

`docs/PROJECT_MEMORY.md` は開発ドキュメントではなく、  
AIセッション間でコンテキストを継承するための **Knowledge Asset** である。

**理由**  
LLMはセッションをまたいで記憶を保持しない。  
Project Memoryを Gitで管理することで、  
どのAIセッションでも同じ設計コンテキストから作業を開始できる。

## D005 — Sprint の完了条件は「能力の獲得」である

**日付**: 2026-06-27
**状態**: 採用

Sprint の終了条件を「コードが完成した」ではなく「新しい能力が獲得された」と定義する。

**理由**
コードの完成は手段であり、目的ではない。
テストが通ることと、能力が存在することは別のことだ。
「ファイルが作成された」と「システムが記憶できる」は、まったく異なる事実である。

能力として定義することで:
- Sprint の成果を人間が観察・検証できる
- 次の Sprint が前の Sprint の能力を前提にできる
- 「何のために作っているか」がコードレベルで明確になる

**能力の定義条件**
1. 名前がある（動詞で表現できる）
2. コードを読まずに示せる
3. その Sprint の前には不可能で、後には可能になっている
4. 他の能力の土台になりうる

**過去 Sprint への適用**

| Sprint | コードの完成 | 獲得した能力 |
|--------|------------|------------|
| Sprint 1 | Foundation構築 | **構築できる** |
| Sprint 2 | Discovery Domain実装 | **発見できる** |
| Sprint 3 | Knowledge Foundation実装 | **記憶できる** |
| Sprint 4 | Semantic Memory実装 | **想起できる** |
| Sprint 5 | Reasoning Foundation実装 | **問いを立てる** |

**影響**
すべての Sprint 計画・レビュー・完了判定はこの定義に従う。
`ROADMAP.md` の各フェーズは能力の名前で記述される。

## D004 — Mock First, AI Later

**日付**: 2026-06-26  
**状態**: 採用

AIを呼ばずにドメインフローを完成させ、  
その後 `ResearchResultPort` の AI実装で差し替える。

**理由**  
AIなしでビジネスロジックが正しく動くことを先に証明する。  
テストコストを抑え、AI呼び出しのレイテンシなしで開発サイクルを高速化する。
